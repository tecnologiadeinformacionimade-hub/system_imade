Introducción y Arquitectura General
Origen del proceso: cómo nace la solicitud que luego aprueba o rechaza el gestor

El flujo Cliente Apertura automatiza el registro y gestión de solicitudes de apertura de código de cliente dentro del ecosistema de Grupo IMADE. Se activa cuando un usuario envía una respuesta en Microsoft Forms y ejecuta una cadena de pasos que van desde la recopilación de datos hasta el adjunto de la documentación respaldatoria, dejando el ítem listo en estado Pendiente para que los Flujos 1 (Aprobación) y 2 (Rechazo) lo procesen.

El flujo está alojado en el entorno de Power Automate de Microsoft 365 y opera mediante la conexión autenticada de tecnologiadeinformacion@grupoimade.com. Todos los datos se almacenan en la lista de SharePoint Clientes, dentro del sitio Apertura Clientes — la misma lista de origen que disparan los Flujos 1 y 2.

Procesamiento de Documentos Adjuntos
Patrón repetido 5 veces: Parse JSON + Apply to each, para DPI, RTU, Foto, Patente y Luz

Cada uno de los cinco campos de carga de archivos del formulario (DPI, RTU, Foto del Negocio, Patente de Comercio y Recibo de Luz) sigue exactamente el mismo patrón de dos acciones: primero un Parse JSON que estructura el array de archivos adjuntos, y luego un Apply to each que descarga y adjunta cada archivo al ítem de SharePoint creado en el paso 3.3.

6–7PROCESAR DPIDPI Ambos lados, DPI, DPI (3 tokens)Apply to each — Get file content + Add attachment
8–9PROCESAR RTURTU, RTU, RTU (3 tokens)Apply to each 1 — Get file content 1 + Add attachment 1
10PROCESAR FOTOFoto del Negocio ×3 tokensApply to each 2 — Get file content 2 + Add attachment 2

Configuración interna de cada bucle (idéntica en los 5)
SubacciónConfiguración
Select an outputBody (salida del Parse JSON correspondiente)
Get file contentConector OneDrive/SharePoint Files · File = Body id del ítem actual del bucle
Add attachmentConector SharePoint · List Name = Clientes · Id = ID del Create item (paso 3.3) · File Name = Body name · File Content = salida de Get file content
Para crear cada nuevo bloque, copie el par PROCESAR [DOCUMENTO] + Apply to each existente, reemplace los tokens del campo Content por los del nuevo campo del formulario y renombre la acción. El schema JSON no cambia.
Renombre todas las acciones con nombres descriptivos en español (PROCESAR DPI, PROCESAR RTU, etc.). Power Automate usa estos nombres como identificadores internos en expresiones, por lo que cambiarlos después puede romper referencias dinámicas.

Introducción y Arquitectura General
Descripción del sistema y listas SharePoint involucradas

Este documento describe la implementación técnica de dos flujos automatizados en Microsoft Power Automate, cuyo propósito es gestionar el ciclo de vida de solicitudes de apertura de clientes (contado/crédito) almacenadas en SharePoint Online.

Cuando un usuario gestor modifica el campo Estado de un ítem en la lista Clientes, el sistema evalúa automáticamente si el estado es Aprobado o Rechazado, y ejecuta la cadena de acciones correspondiente: notificación por correo, creación del registro en la lista destino, copia de adjuntos y eliminación del ítem de la lista de trabajo.

ProblemaCausa probable y solución
El flujo no se dispara al modificar un ítemVerificar que la conexión del trigger apunta a la lista Clientes correcta. Revisar que la cuenta de servicio tenga permisos de lectura. Comprobar que el flujo esté activado (estado On).
La condición siempre va a FalseEl valor del campo Estado no coincide exactamente con Aprobado o Rechazado (mayúsculas/minúsculas). Revisar las opciones de la columna Choice en SharePoint y ajustar el valor literal.
El correo no se envíaLa columna Solicitante no es de tipo Person or Group, o el usuario no tiene correo asignado en Azure AD. Verificar la conexión del conector Outlook.
Create item falla con error de campoAlguna columna mapeada no existe en la lista destino. Revisar que todas las columnas estén creadas con los mismos nombres internos.
Los adjuntos no se copianEl bucle Apply to each itera sobre un array vacío (el ítem no tenía adjuntos). Si hay adjuntos y no se copian, verificar que Get attachment content usa el campo Id del ítem del bucle (no el ID del trigger).
El ítem no se elimina de ClientesDelete item usa el campo ID del trigger. Si falla, revisar los errores en el historial de ejecución del flujo en Power Automate.
Ambos flujos se ejecutan para el mismo ítemCorrecto: ambos flujos escuchan la misma lista. La condición de cada uno filtra según Estado. Si Estado = Aprobado, solo Flujo 1 procede; si Estado = Rechazado, solo Flujo 2 procede.
7.1 Verificación post-implementación
1
Prueba Flujo 1
Crear un ítem de prueba en la lista Clientes con Estado = Aprobado y verificar que el Flujo 1 se ejecuta completamente.
2
Verificar lista Aprobados
Confirmar que el ítem aparece en la lista Aprobados con todos sus campos y adjuntos.
3
Verificar correo
Confirmar que el correo llegó al Solicitante.
4
Verificar eliminación
Confirmar que el ítem fue eliminado de la lista Clientes.
5
Prueba Flujo 2
Repetir el proceso con Estado = Rechazado para el Flujo 2.
6
Revisar historial
Revisar el historial de ejecución en Power Automate (Monitor > Cloud flow activity) para confirmar 0 errores.
