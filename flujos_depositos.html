Confirmación de Depósitos Bancarios
Flujo automatizado en Power Automate que procesa las boletas de depósito enviadas por clientes a través de Microsoft Forms, registrándolas en SharePoint y notificando a los equipos correspondientes vía correo electrónico.
Power Automate Microsoft Forms SharePoint Activo 7 Pasos Outlook 365
📋
Resumen del flujo
Objetivo, herramientas y conexiones involucradas
Este flujo automatiza el proceso de recepción y registro de boletas de depósito bancario. Cuando un cliente o colaborador llena el formulario "Confirmación de depósitos bancarios" en Microsoft Forms adjuntando su comprobante, el flujo se ejecuta automáticamente para notificar al equipo contable, registrar la solicitud en SharePoint y confirmarle al remitente que su boleta fue recibida.
Cuenta de servicio: El flujo opera conectado a tecnologiadeinformacion@grupoimade.com y el sitio de SharePoint utilizado es grupoimade.sharepoint.com/sites/ConfirmacindeDepositosBancarios/.

Análisis del archivo JSON
Data Operations · Parsear adjuntos del formulario
El campo de adjuntos de Microsoft Forms llega como una cadena de texto en formato JSON. Esta acción la convierte en un array de objetos que el bucle siguiente puede recorrer, uno por cada archivo adjunto.
Esquema utilizado
El esquema define la estructura esperada de cada objeto dentro del array:
PropiedadTipoDescripción

linkstringURL de descarga del archivo
idstringIdentificador único del archivo en Forms
typeobjectTipo MIME del archivo
sizeintegerTamaño del archivo en bytes
referenceIdstringID de referencia en OneDrive
El campo de entrada (Content) es la variable dinámica "Ingrese su depósito" proveniente de los detalles del formulario.

Estructura del sitio ConfirmacindeDepositosBancarios
El sitio SharePoint contiene tres listas que representan los distintos estados del ciclo de vida de una boleta. El flujo automatizado solo escribe en Depositos Bancarios; las otras dos listas son actualizadas manualmente por el equipo contable.
📥

● Pendiente
Lista principal donde el flujo registra cada nueva boleta. Estado inicial: Pendiente.
✅

● Boleta Confirmada
Registro de todas las boletas validadas por el equipo contable.
❌
Boletas Rechazadas

Boletas que no pudieron ser confirmadas, con observación del motivo.
Columnas de las listas
Todas las listas comparten las siguientes columnas:
Datos adjuntos Observaciones Estado Correo Electronico Link de imagen Fecha y Hora Comentarios
La columna Fecha y Hora en la lista "Boletas Confirmadas" muestra "Información necesaria" para ítems migrados o creados manualmente, ya que ese campo es requerido en el flujo automatizado pero opcional en carga manual.

Resumen — Boleta Confirmada
Flujo secundario · Confirmación contable y notificación al cliente
Este flujo se activa cuando el equipo contable actualiza el Estado de un ítem en la lista Depositos Bancarios a Boleta confirmada. Su objetivo es trasladar el registro a la lista Boletas Confirmadas, copiar los adjuntos, notificar al cliente por correo electrónico y limpiar el ítem original.
Este flujo es el segundo eslabón del ciclo: actúa justo después de que el contador revisa y aprueba la boleta, cerrando el proceso con la notificación al cliente.

Resumen — Boleta Rechazada
Flujo de rechazo contable · Notificación y archivado
Este flujo se activa cuando el equipo contable actualiza el Estado de un ítem en la lista Depositos Bancarios a Boleta rechazada. Su objetivo es notificar al cliente del rechazo, trasladar el registro a la lista Boletas Rechazadas con las observaciones del motivo, copiar los adjuntos y limpiar el ítem original de la lista de pendientes.
Este flujo es el tercer eslabón del ciclo: actúa cuando el contador identifica inconsistencias en la boleta. El cliente es notificado para que reenvíe la boleta correcta a la brevedad posible.

1
Disparador — When an item is created or modified
Trigger de SharePoint · Lista Depositos Bancarios
Este disparador monitorea constantemente la lista Depositos Bancarios en el sitio de SharePoint de Confirmación de Depósitos Bancarios. Se ejecuta automáticamente cada vez que se crea o modifica cualquier ítem en esa lista, lo que permite reaccionar al momento exacto en que el equipo contable cambia el estado a "Boleta rechazada".
ParámetroValor
Site Addressgrupoimade.sharepoint.com/sites/ConfirmacindeDepositosBancarios
List NameDepositos Bancarios
Connected astecnologiadeinformacion@grupoimade.com
El disparador se activa para cualquier creación o modificación en la lista. Por eso es imprescindible la condición del paso 2 para filtrar únicamente los ítems con estado "Boleta rechazada" y evitar ejecuciones no deseadas.
2
Condición — Filtro de estado
Verifica que Estado/Value = "Boleta rechazada"
Esta condición es la puerta de entrada del flujo. Solo cuando el campo Estado del ítem tiene el valor exacto Boleta rechazada se ejecuta la rama True con los 6 pasos de procesamiento. En cualquier otro caso, la rama False (0 acciones) termina el flujo silenciosamente.

Campo evaluadoEstado/Value — valor del campo de tipo Choice
Operador de comparaciónis equal to

Rama True6 acciones: correo + crear + get adjuntos + foreach + eliminar
Rama False0 acciones — el flujo termina sin hacer nada
El valor "Boleta rechazada" debe coincidir exactamente con la opción definida en la columna Estado de la lista de SharePoint. Si se cambia el nombre de la opción en la lista, también debe actualizarse aquí.

Delete item — Limpieza de Depositos Bancarios
Elimina el ítem rechazado de la lista de pendientes
Una vez que el ítem rechazado ha sido copiado a Boletas Rechazadas con todos sus adjuntos, y el cliente ha sido notificado, este paso elimina el ítem original de la lista Depositos Bancarios. Esto mantiene la lista de pendientes limpia, con únicamente las boletas que aún están esperando revisión contable.

IdDinámica — ID del ítem disparador
La eliminación es permanente e irreversible. El registro histórico queda únicamente en la lista Boletas Rechazadas. Asegúrate de que los adjuntos se hayan copiado correctamente antes de que este paso se ejecute. Si el flujo falla en el paso 6, el ítem no se eliminará ya que este paso no se alcanzaría.
Tras este paso, el ciclo de vida de la boleta rechazada queda completo: el cliente fue notificado para que reenvíe, el registro quedó archivado en Boletas Rechazadas con observaciones y adjuntos, y la lista de pendientes queda despejada para nuevas boletas.
