Creación e Inicialización de Tickets
Este flujo automatiza la captura de solicitudes de soporte enviadas a través de Microsoft Forms, registrándolas de forma centralizada en SharePoint y generando alertas inmediatas al equipo de TI.

Paso 2 — Obtención de Detalles e Inicialización de Variables
Extracción de respuestas y variables internas

Agregue la acción Get response details para extraer las respuestas dinámicas de la solicitud. Inmediatamente después, inicialice las variables necesarias para el control interno de flujos:

JSON
// Inicialización de variables de control de archivos adjuntos { "Name": "varAdjuntos", "Type": "Array", "Value": [] }

Paso 3 — Consulta de Consecutivo del Día y Ordenamiento
Get items y organización por fecha

Para mantener una correlación idónea, inserte un bloque Get items apuntando a la lista de SharePoint. Ordene los registros de forma descendente para capturar el último número de caso creado hoy:

Propiedad AvanzadaConfiguración
Filter QueryCreated ge formatDateTime(utcNow(), 'yyyy-MM-dd')
Order ByConsecutivo Dia desc
Top Count1
5
Paso 4 — Evaluación, Notificación y Procesamiento
Creación en SharePoint y alertas finales

Inserte una estructura condicional. Si se cumple el orden dinámico, ejecute Create item para insertar el registro en la lista de soporte oficial. Seguido, añada las bifurcaciones automatizadas de notificación (Email y Push).

Flujo 1 — ID Ticket Update Mail Proceso (Parámetros del Trigger)
Activación en bandeja de entrada

Este flujo se dispara al recibir un nuevo correo electrónico. Se aplican filtros avanzados directamente en el trigger para optimizar ejecuciones: solo se activa si el asunto contiene la frase exacta "Ticket generado" y llega específicamente a la carpeta de la Bandeja de entrada.

3
Flujo 2 — ID Ticket Update Mail Proceso (Vista General)
Arquitectura del proceso principal

Estructura general del proceso principal de actualización. Tras el trigger, se incluye la obtención de ítems (Get items) desde la lista de SharePoint, un ciclo iterativo (Apply to each) que evalúa los resultados, y una condición que bifurca hacia la actualización del ítem en caso afirmativo.

4
Flujo 3 — ID Ticket Update Mail Inicio (Lógica de la Condición)
Validación y coincidencia de ticket

En la evaluación lógica (Condition) del ciclo, el flujo comprueba que el elemento que se está analizando coincida con el correo. Específicamente, verifica que el campo del correo entrante "Subject" (Asunto) contenga el valor dinámico "NumerodeTicket" traído desde la base de datos de SharePoint.

5
Flujo 4 — ID Ticket Update Mail Record a Proces (Agente)
Interacción directa de TI

Variante del flujo de actualización orientada a la interacción directa del Agente de TI. Su arquitectura refleja el modelo de sincronización para mapear la trazabilidad entre el hilo de respuesta del agente y el seguimiento del ticket original.

6
Flujo 5 — ID Ticket Update Mail Record a Proces
Consolidación general de estados

Flujo de consolidación general para mantener la bitácora del proceso, asegurando la actualización y consistencia de los estados del ítem en SharePoint de forma automatizada y sin intervención manual redundante.
